# Project Context Builder

既存リポジトリを自動解析し、AI コーディングアシスタントが即戦力になれるコンテキストファイルを
対話式に生成するスキル。ソースコードから推論できることは推論し、
ユーザーへの質問は「自動推論では決められない選択」だけに絞る。

---

## コンテキストファイルとは

生成前にユーザーへ必ず説明する：

```
AI コーディングアシスタント向けの「プロジェクト説明書」です。

これがあると AI は：
- プロジェクトの目的・技術スタックを最初から把握できる
- チームのコーディング規約・命名規則を守れる
- 使ってはいけないコマンドや注意点を事前に知れる
- コミットメッセージや PR の書き方を統一できる

ツールによって読み込むファイルが異なります：

| ツール              | ファイル                              |
|---------------------|---------------------------------------|
| Claude Code         | CLAUDE.md（プロジェクトルート）       |
| GitHub Copilot      | .github/copilot-instructions.md       |
| Cursor / Gemini CLI / その他 | AGENTS.md（オープン標準）   |

まずどのツール向けに生成しますか？
```

---

## 生成フロー

**必ず1問ずつ順番に質問すること。次の質問は前の回答を受け取ってから行う。**

### Phase 0: ソースコード自動解析（質問なし）

最初にリポジトリを自動スキャンする。
**ユーザーへの質問は不要。読めた情報だけで推論し、サマリーを提示する。**

#### Step 1: ディレクトリ構成の把握

```bash
find . -maxdepth 3 -type f \
  | grep -v node_modules | grep -v .git | grep -v vendor \
  | sort
```

フレームワーク・言語・主要ディレクトリ構成を推定する。

#### Step 2: パッケージ定義ファイルを読む

| ファイル | 読み取る情報 |
|----------|-------------|
| `composer.json` | PHP バージョン・フレームワーク・主要パッケージ |
| `package.json` | フロントフレームワーク・スクリプト（dev/build/test） |
| `requirements.txt` / `pyproject.toml` | Python パッケージ・バージョン |
| `go.mod` / `Cargo.toml` / `Gemfile` | 他言語のスタック |
| `pom.xml` / `build.gradle` | Java/Kotlin スタック |

#### Step 3: 開発環境ファイルを読む

| ファイル | 読み取る情報 |
|----------|-------------|
| `docker-compose.yml` | サービス名・ポート・DB・キャッシュ |
| `Makefile` | よく使うコマンド一覧 |
| `.env.example` | 必要な環境変数 |
| `README.md` | プロジェクト概要・セットアップ手順 |
| `.github/workflows/` | CI コマンド（テスト・lint・ビルド） |

#### Step 4: 既存の AI 設定ファイルを確認

| ファイル | 処理 |
|----------|------|
| `CLAUDE.md` | 既存あり → 上書き or 更新かを確認 |
| `AGENTS.md` | 内容を把握して参考にする |
| `.github/copilot-instructions.md` | 内容を把握して参考にする |
| `.cursor/rules` / `.clinerules` | 内容を把握して参考にする |

#### Step 5: コーディング規約ファイルを確認（内容は転記しない）

**⚠️ 規約ファイルの内容をコンテキストファイルに転記しない。参照パスだけを書く。**

| ファイル | 用途 |
|----------|------|
| `phpcs.xml` / `.phpcs.xml` | PHP 規約 |
| `eslint.config.js` / `.eslintrc*` | JS/TS 規約 |
| `.prettierrc*` | フォーマット設定 |
| `.editorconfig` | インデント・改行 |
| `pyproject.toml` の `[tool.ruff]` | Python lint |

#### Step 6: 解析サマリーの提示

```
ソースコードを確認しました。以下のように把握しました：

**プロジェクト概要**
（README や構成から読み取った概要を 1〜2 文で）

**技術スタック**
- 言語：PHP 8.3 / JavaScript (Node 20)
- フレームワーク：Laravel 11 / Vue.js 3 + Vite
- DB：MySQL 8 / Redis
- インフラ：Docker Compose / AWS

**よく使うコマンド（推定）**
- 起動：docker compose up -d
- テスト：php artisan test / npm run test
- Lint：./vendor/bin/phpcs / npm run lint

**規約ファイル**
- phpcs.xml ✅  eslint.config.js ✅  .prettierrc ✅

認識に誤りや補足があれば教えてください。問題なければ次に進みます。
```

ユーザーが確認したら Phase 2 へ進む。

---

### Phase 2: 選択が必要な項目を1問ずつ確認

自動推論では決められない「チームの意思決定」だけを聞く。

---

**Q1: 対象ツールの選択**

```
どの AI ツール向けに生成しますか？

1. Claude Code          → CLAUDE.md
2. GitHub Copilot       → .github/copilot-instructions.md
3. その他（Cursor / Gemini CLI / Codex など）→ AGENTS.md（オープン標準）
4. 複数ツールまとめて生成
```

Q1 の回答に応じて、以降の質問文・生成ファイルを切り替える。
複数選択の場合は共通内容を 1 つ作成し、ツール固有セクションを追加する。

---

**Q2: 記述言語**

```
コンテキストファイルはどの言語で書きますか？

1. 日本語（技術用語は英語のまま）
2. English
3. 両言語（日本語メインで英語補足）
```

---

**Q3: コミットメッセージのルール**

```
コミットメッセージのフォーマットはありますか？

1. Conventional Commits（feat / fix / chore / docs など）
2. 日本語で自由記述
3. 独自ルールがある（次で教えてください）
4. 特に決まっていない
```

「3. 独自ルール」の場合のみ追加質問：

```
コミットメッセージのフォーマットを教えてください。
（例：「[JIRA-123] 修正内容」「WIP: 作業中」など）
```

---

**Q4: ブランチ戦略**

```
ブランチの命名ルールはありますか？

1. feature/xxx, fix/xxx, hotfix/xxx（GitFlow 系）
2. {issue番号}-{説明}（例：123-add-login）
3. 特に決まっていない
4. 独自ルールがある（教えてください）
```

---

**Q5: やってはいけないこと（禁止事項）**

```
AI に絶対やらせたくないことはありますか？

よくある例：
- 本番 DB への直接接続・操作
- .env ファイルの上書き
- main / master への直接 push
- 特定コマンドの実行（rm -rf など）

「特になし」でも OK です。
```

---

**Q6: テスト方針**

```
テストについてのルールはありますか？

1. 実装と同時にテストも書く（TDD）
2. 実装後にテストを書く
3. テストは任意
4. 特に決まっていない
```

全問回答が終わったら生成フェーズへ進む。

## ファイル別の生成ルール

### 共通テンプレート構成

```markdown
# {ファイル名 / プロジェクト名}

## プロジェクト概要
（解析で得た概要 + ユーザー補足）

## 技術スタック
（解析で得たスタック一覧）

## セットアップ

### 前提条件
（Docker / Node / PHP バージョンなど）

### 起動手順
（解析で得たコマンド）

## よく使うコマンド

| コマンド | 説明 |
|----------|------|
| `{コマンド}` | {説明} |

## コーディング規約

規約ファイルを必ず参照してください（内容はここに転記しない）：
- {規約ファイルのパスを列挙（存在するもののみ）}

## コミットメッセージ

{Q3 の回答に基づくフォーマットと具体例}

## ブランチ戦略

{Q4 の回答に基づく説明}

## ⚠️ やってはいけないこと

{Q5 の回答に基づく禁止事項（「特になし」の場合はセクション省略）}

## テスト方針

{Q6 の回答に基づく説明}

## ディレクトリ構成

{主要ディレクトリの役割を簡潔に。深さ 2〜3 まで}

## 環境変数

{.env.example から読み取った必須の環境変数と説明}
（機密値は記載しない。.env.example を参照するよう誘導）
```

### ツール別の差分

**Claude Code（CLAUDE.md）**
- セクション名は日本語 or 英語（Q2 に従う）
- `## よく使うコマンド` は必須（Claude Code が頻繁に参照する）
- グローバル設定は `~/.claude/CLAUDE.md` に別途案内

**GitHub Copilot（.github/copilot-instructions.md）**
- Copilot は英語の方が精度が高いことを案内する
- `applyTo` フロントマターは不要（instructions ファイルは全体に適用）
- PR テンプレート（`.github/pull_request_template.md`）との連携を提案

**AGENTS.md（その他ツール）**
- オープン標準フォーマットに準拠
- サブディレクトリへの配置も案内（ディレクトリ単位で上書き可能）
- 複数ツールで共用することを明記

### 記述の共通ルール

- **規約ファイルの内容を転記しない**（パスを参照するだけ）
- **機密情報を含めない**（API キー・パスワードは絶対に書かない）
- **コマンドは実際に動くものだけ書く**（推定では書かない）
- **不明な情報はプレースホルダーで示す**（`# TODO: 確認が必要` など）
- 技術用語（コマンド名・ファイル名・ライブラリ名）は言語によらず英語のまま

## 生成後の処理

### Step 1: ファイル出力

生成ファイルを所定のパスに生成し `present_files` で提示する。

### Step 2: サブディレクトリ版の提案

モノレポやフロント・バック分離構成の場合に提案する：

```
バックエンドとフロントエンドでスタックが大きく異なります。
サブディレクトリ用のファイルも作成しますか？

1. backend/{ファイル名}
2. frontend/{ファイル名}
3. 両方
4. ルートのみで OK
```

### Step 3: 次のステップを案内

```
## 完了 ✅

{生成したファイルパス} を生成しました。

  git add {ファイルパス}
  git commit -m "docs: add AI context file"
  git push

## さらに AI の理解を深めるには
```

Claude Code 向けに生成した場合 → AGENTS.md の作成を提案
Copilot 向けに生成した場合 → CLAUDE.md の作成も提案
AGENTS.md を生成した場合 → 各ツール固有ファイルへのシンボリックリンクを提案

---

## 注意事項

- ファイル名に特定の AI ツール名を含む場合（CLAUDE.md など）は、
  そのツール専用であることをユーザーに伝える
- 複数ツールを使うチームには AGENTS.md を基本として
  各ツール固有ファイルからシンボリックリンクを張る方法も案内する
- 機密情報（API キー・パスワード・接続情報）は絶対に含めない
- スタック変更・ルール変更のたびに更新が必要であることを伝える
